# Phase P2b — `internal/expr` Restricted Expression Engine (parity, behind flag)

> **For agentic workers:** REQUIRED SUB-SKILL: superpowers:subagent-driven-development (or superpowers:test-driven-development per task). Steps use `- [ ]` checkboxes. No production code before a red test.

**Goal:** Build a stdlib-only restricted-grammar expression engine (`internal/expr`) that resolves n8n-style `{{ … }}` templates over the P2a items model, wired into the worker as a **parity alternative** to `substituteVars` behind a flag — NOT the sole execution channel. Security-critical: the cross-tenant scoping (S-2) and secret-channel (S-3) invariants must be correct from the first commit.

**Architecture:** Three leaf-first layers — (1) pure engine `internal/expr` (no DB/worker import): tokenizer → parser → evaluator for the restricted grammar; `Resolve(template, Context) (string, error)`, `Context{Self []Item, NodeByID func(id)([]Item,error), ItemIdx int}`, with `expr.Item` mirroring `worker.Item` so the package stays a leaf. (2) worker-side scoped `NodeByID` resolver (`exprNodeResolver`) enforcing the **S-2 contract** — resolves only ids in the executing todo's DIRECT `dependsOn` set, scoped by `project_id` (rejects out-of-set + cross-project), reusing P2a `itemsForDep`. (3) behind-flag parity wiring (`Config.ExprParity bool`): computes the expr result alongside `substituteVars`, asserts/logs equivalence, NEVER feeds expr downstream (cut-over is P3). The secret channel is untouched: secrets resolve ONCE (worker.go ~1845, `secretRefRe` + `Secrets.Resolve`, orgID from `OrgIDForProject(c.projectID)`) BEFORE either `{{name}}` channel; the expr grammar treats `secret:` as a non-matching literal emitted verbatim.

**Tech Stack:** Go stdlib only inside `internal/expr` (NO goja/starlark/JS VM). `internal/scriptengine` (Starlark) is the SEPARATE escape-hatch; P2b does not import/touch it (S-6 re-asserted). DB-backed resolver tests use Postgres fresh DB `-p 1`.

## Real-code findings (from planning, grounded)
- **Secret resolution today** (runCustomHTTP worker.go:1813-1903): per header value, order is strict — (1) `orgID,_ := w.cfg.Projects.OrgIDForProject(ctx, c.projectID)` (orgID ONLY from trusted run context, never input JSON); (2) `secretRefRe.ReplaceAllStringFunc` resolves `{{secret:NAME}}` → `Secrets.Resolve(ctx, orgID, name)`, sets `secretBearing`; (3) `substituteVars` runs SECOND on the secret-resolved result — so a `{{secret:X}}` arriving via a `{{name}}` upstream value is NEVER resolved (secret pass already done); (4) body forbids secrets entirely (post-substitute match → `errRequestFailed`); (5) `secretBearing && !AllowResponseBody` → store only `{"status":N}`. `secretRefRe = \{\{\s*secret:([A-Za-z0-9_\-]+)\s*\}\}`. **P2b boundary:** expr engine runs as a PEER to `substituteVars` (the `{{name}}` channel only), AFTER the secret pass; treats `secret:` as literal; touches none of `secretRefRe`/`Secrets.Resolve`/`OrgIDForProject`/`{status}` guard.
- **dependsOn for S-2:** `loadInputs` reads `SELECT depends_on FROM todos WHERE id=$1` (direct deps, `pq.StringArray`); planner `depSet` (planner.go:279-286) is also direct-deps-only. `loadInputs` only materializes direct deps' items. **A1 decision: direct-deps + project-scope suffices** (a transitive `$node` id would be a dead reference — no items loaded). Cross-project defense: resolver loads each candidate dep's `project_id`, rejects any `!= c.projectID`.
- **Reuse vs parallel:** `substituteVars` (worker.go:1999) = the channel P2b PARALLELS (unchanged). `resolveOutputText`/`resolveVariables` = legacy seam, stays. `loadInputs`/`itemsForDep` (P2a) = items source (`ctx.Self`). No existing restricted-grammar engine — `internal/expr` is net-new.

## Ambiguity-calls
- **A1 (direct vs transitive depSet) — RATIFIED by security review as a deliberate fail-closed ceiling (F5):** spec §3.4 says "传递 dependsOn 祖先集" (transitive) but the reused `PlanCustom` depSet + `loadInputs` are direct-deps-only, and `loadInputs` materializes only direct deps' items. Direct-deps is **strictly safer** — a `$node["transitive-ancestor"]` is REJECTED (out-of-set), denied by default, not silently read. **Transitive reads are denied BY DESIGN, not a temporary limitation:** any future need for ancestor data must add an explicit edge (making it a direct dep), NEVER widen `NodeByID` to a transitive closure. Contrast with `resolveVariables` (worker.go:1732, bare `WHERE id=$1`, trusted-idMap model) — the new resolver must NOT inherit that trust model, and a P3 cutover replacing `resolveVariables` with expr must carry the project gate forward (F8).
- **★ P2b runtime coverage (F2):** `$node["id"]` has **no live producer in P2b** — the planner emits a `variables` list (`sourceTodoId`), not `$node` expression tokens; the `$node` token rewrite is P3 (spec §211). So the parity probe's `toExprTemplate` produces only `$json` refs, and `exprNodeResolver`'s S-2 gate is reachable ONLY from its tests in P2b. The resolver ships AHEAD of its consumer; S-2 is proven by the DB-backed two-project test, not runtime — hence that test is MANDATORY and real-DB (not stubbed). `$binary` items are likewise not produced in P2b (itemsForDep `asset:` → nil); `$binary.x` is grammar-only until P3 (F7).
- **A2 (flag mechanism):** `Config.ExprParity bool` (additive, default false) gating a parity-only path that computes+compares but does NOT change the downstream value. Cut-over to expr-as-channel is P3.
- **A3 (helpers):** minimal whitelist `$now` (RFC3339 UTC, documented deterministic-enough), `JSON.stringify`, `.toLowerCase()` per §3.4.
